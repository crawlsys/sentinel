### Added

- **M2.6 federation deprecation/migration directives ‚Äî `deprecated` + `override`**: ports Apollo Federation's `@deprecated` (with reason string) and `@override` (replacement declaration) to step configs. `WorkflowStep` gains `deprecated: Option<String>` and `r#override: Option<String>`, both `#[serde(default)]` so M2.5 configs load unchanged. Override grammar accepts `"phase.step_id"` (same skill ‚Äî the common case) and `"skill.phase.step_id"` (cross-skill, when one skill takes over capability previously owned by another). `sentinel federation compose` (and its `check` companion) gain a deprecation pass: deprecated steps surface as warnings (not errors ‚Äî they still work), dangling override targets surface as hard errors (you can't replace a step that isn't there), and override targets that aren't themselves marked `deprecated` surface as warnings (encourages disciplined migration paths). New helper `parse_override_target()` handles both grammars. 12 new tests (3 in `config.rs` for round-trip, 6 cross-skill compose tests covering deprecated warning / dangling override / target-not-deprecated / clean migration / malformed grammar / cross-skill resolution, 3 helper-fn unit tests). 1221/1221 workspace tests green.

- **M2.5 federation directives ‚Äî `provides`, `requires`, `external`, `inaccessible` on step configs**: ports Apollo Federation's `@provides` / `@requires` / `@external` / `@inaccessible` directives to the federated step namespace. `WorkflowStep` in `sentinel-domain` gains four optional fields with `#[serde(default)]` so every existing TOML loads unchanged. `StepToml` in `sentinel-infrastructure` threads them through the loader. `sentinel federation compose` (and its `check` companion) gain a second pass after the per-skill internal checks: it builds two indexes from the union of all loaded skills (artifact providers + step coordinates) and walks every step's `requires` / `external` declarations, raising hard errors when a `requires` has no producer in the supergraph or an `external` reference is dangling/malformed. `inaccessible` steps still participate in the provides graph ‚Äî they're internal-only from the router's perspective (M7 will filter on the flag), not from the federation contract's. 8 new tests (3 in `config.rs` for round-trip + defaults, 5 in `federation_cmd.rs` for cross-skill resolution: clean cross-skill chain, missing producer, dangling external, malformed external, inaccessible-still-counts). 1209/1209 workspace tests green.

### Fixed

- **`project_hash` now canonicalizes worktree paths to their parent repo ‚Äî task durability survives worktree-switches**: the bug task #70 (orig_id) and bloat task #72 (orig_id) had the same root cause. Six hook files (`task_persist`, `task_rehydrate`, `memory_extract`, `session_index`, `session_init`, `todo_interceptor`) each carried a copy-paste of `fn project_hash(cwd) -> sha256(cwd.as_bytes())[..4]`. Hashing the cwd verbatim meant `/repo` and `/repo/.claude/worktrees/feat-x` produced different hashes, so `task_persist` wrote one bucket while `task_rehydrate` looked up another every time the user crossed a worktree boundary ‚Äî tasks silently disappeared. New shared helpers in `crates/sentinel-application/src/hooks/mod.rs`: `canonical_project_cwd(cwd)` strips the literal segment `/.claude/worktrees/` (and the `\.claude\worktrees\` Windows variant) plus everything after it, collapsing any worktree to its repo root; `project_hash(cwd)` uses the canonical form. All 6 callers now delegate to `super::project_hash`. Dead `sha2::{Digest, Sha256}` imports + the local `hex_encode` helper removed from `task_rehydrate.rs` and `todo_interceptor.rs`. 6 new unit tests pin the worktree-collapse invariant: forward-slash collapse, backslash collapse, deep subpath collapse, non-worktree paths unchanged, sentinel main + sentinel worktree produce identical hashes, different repos still produce different hashes. The 390 orphan project-hash directories accumulated under the broken behavior remain on disk ‚Äî cleanup is filed as a separate task. 798 sentinel-application library tests pass; release build clean; engine staged via `sentinel stage` for hot-swap on next session.

- **`memory_verify` re-wired to PreCompact ‚Äî completes the relocation tracked as a follow-up in the previous hooks.toml fix**: the module had been orphaned on disk since the SessionStart removal (Qdrant scroll + per-claim Cerebras verification was blocking startup 5‚Äì20s on a cold cache). (2) `crates/sentinel-cli/src/hook_cmd.rs` ‚Äî `memory_verify::process` now invoked from the PreCompact dispatcher branch alongside `pre_compact::process` and `session_index::process`, with an inline comment recording the relocation rationale. (3) `config/hooks.toml` ‚Äî replaced the "orphaned, kept on disk" comment with a registration entry under PreCompact (`depends_on = ["pre-compact"]`, `has_api_call = true`); the inline comment now points readers to the new home. 792 sentinel-application library tests pass; release build clean.

The orphaned `memory_verify` Rust module is left in place for a planned PreCompact relocation, tracked as a follow-up. 1075 workspace tests pass.

- **Flaky `api::memory::tests::daemon_url_falls_back_to_default_when_unset` race**: three tests in `crates/sentinel-cli/src/api/memory.rs` (`daemon_url_uses_env_override_when_set`, `daemon_url_falls_back_to_default_when_unset`, `daemon_unreachable_returns_503`) all read+mutate `SENTINEL_MEMORY_DAEMON_URL`. The previous comment claimed the override test was "the only env-touching test" ‚Äî wrong, the unreachable test was added later without the same treatment. ### Changed

- **`task_rehydrate` is mode-aware: auto-rehydrates in Autopilot, asks first in Planned**: prior behavior was "always ASK FIRST regardless of mode" which made every Autopilot session pay an interruption tax to recover its task queue from the previous session. The hook context already exposes `autopilot_enabled()` (reads `SENTINEL_AUTOPILOT`); `task_rehydrate.rs` now consults it and emits one of four instruction variants (autopilot+blocking, autopilot+flat, planned+blocking, planned+flat). Autopilot variants direct immediate `TaskCreate` (and `TaskUpdate(addBlockedBy)` for blocking chains) followed by a one-sentence brief; Planned variants preserve the y/n ask-gate. Logic extracted to a pure helper `format_rehydrate_instruction(count, has_blocking, autopilot)` so the four-way branching is unit-tested without FS plumbing ‚Äî 4 new tests pin each variant's invariants (mode tag, action verb, count formatting, mutual-exclusion of mode strings).

- **`tool_usage_gate` now strictly requires `TaskCreate` ‚Äî `TodoWrite` no longer satisfies the gate**: per Gary's `CLAUDE.md` "Required Tool Usage" ¬ß3, the agent-harness `TaskCreate` / `TaskUpdate` (TaskList) tool is mandatory for tracking work, and the dual-acceptance of `TodoWrite` was the slippage point that let agents drift into the lesser fallback. Two changes in `crates/sentinel-cli/src/hook_cmd.rs`: (1) the PostToolUse marker writer no longer fires `mark_task_created` / `mark_task_active` on `TodoWrite` ‚Äî only `TaskCreate` does; (2) the entire `TodoWrite`-with-`in_progress`-payload branch that opportunistically wrote `TASK_ACTIVE_PREFIX` is removed. Deny messages in `tool_usage_gate.rs` updated in two places (Check 2 and Check 4) to specifically name `TaskCreate` and explicitly reject `TodoWrite` with a "NOT accepted ‚Äî Gary's CLAUDE.md mandates the agent-harness TaskList tool" callout, so agents that hit the block know exactly what to use. The signed `verification` override (60s escape hatch) still works as before. New regression test `test_check2_deny_message_rejects_todowrite_and_names_taskcreate` pins both invariants ‚Äî if a future edit drops either the `TaskCreate` mention or the `TodoWrite` rejection, the test fails.

- **`tool_usage_gate` Check 4 reads the persistent task store, not just the sticky marker**: the `TASK_ACTIVE_PREFIX` marker only flips ON ‚Äî it never reflects task completion ‚Äî so once any task was marked active, the marker satisfied the gate forever even after every task completed. New `persistent_store_has_active_task(fs)` reads `~/.claude/sentinel/persistent-tasks/*/tasks.json` (and the legacy `~/.claude/persistent-tasks/*/tasks.json` location during the migration window) and returns `true` only when at least one task currently has `status=in_progress`. Check 4 consults this authoritative source first; the marker is retained as a fast-path/fallback for the rare case the store can't be read (no home dir, malformed JSON). Catches the "all tasks are completed but agent is still editing" drift that motivated the recurring complaint.

### Fixed

- **`git_hygiene` threshold raised + `pre_commit_verification` recognizes more verifiers**: `MAX_UNCOMMITTED_FILES` was 10, which constantly false-fired on routine work in repos with legacy index noise (stale build dirs, in-progress refactors, .gitignore drift) and forced cleanup detours before the actual task could resume. Raised to 150 ‚Äî still catches genuinely out-of-control sessions but no longer blocks normal work. Separately, `pre_commit_verification`'s `TEST_COMMAND_PATTERNS` was missing `cargo check`, `cargo clippy`, `go build`, and `go vet` ‚Äî meaning Rust changes verified via `cargo check` (the most common Rust verifier) would still be blocked at commit because the gate didn't recognize the evidence. Patterns added. Tests updated to compute file counts off `MAX_UNCOMMITTED_FILES` instead of hardcoded 12/15 so they remain correct under any future threshold changes. 964 tests pass.

- **Interceptor false-positives on flag-like text inside commit messages**: `evaluate_git_command` substring-matched the joined args string, so a commit body containing literal `--force` (or `reset --hard`, `filter-branch`, etc.) would block the commit. Real-world reproduction this session: `git commit -m "fix(interceptor): rm -rf orphaned worktree shells on Windows + better cleanup reminder ... worktree remove [--force] path ..."` was rejected as if `--force` were a real flag. Fix: new `evaluate_git_args(&[String])` entrypoint that strips the values of `-m` / `--message` / `-F` / `--file` (both space-separated and `=` forms) before applying substring-match policy. The application layer (`crates/sentinel-application/src/interceptor.rs`) now uses the args-aware entrypoint. The legacy `evaluate_git_command(&str)` stays for backward compat. 11 new unit tests pin the false-positive cases (commit with `--force` / `reset --hard` / `filter-branch` / `--force-with-lease` / `rebase -i` in the message body, long-form `--message`, `=` form, `-F path`) AND verify real `--force`, `push --force`, `reset --hard` are still blocked/confirmed correctly. 762/762 app + 26/26 domain tests still green.

- **Orphaned worktree shells on Windows**: `git worktree remove` on Windows routinely fails to delete the on-disk directory because some process (file watcher, mcp-router, IDE) holds a handle inside the tree. Git itself unregisters the worktree and exits 0, leaving an empty (or near-empty) directory shell behind that `hygiene_reminders` then correctly flags as stale on every prompt. Two-layer fix:
  - `sentinel-git-interceptor` now runs a `post_exec_cleanup` pass after every wrapped git invocation (allow path, confirm path, and `--bypass` path). When the args parse as `worktree remove [--force] <path>` and git returned 0 *and* the on-disk directory still exists, retries `std::fs::remove_dir_all` up to 3 times with 150ms backoff. Failures are logged to stderr and never bubble up ‚Äî git already considered the command successful. NotFound errors are treated as a benign race.
  - `hygiene_reminders` cleanup message now includes a copy-paste `rm -rf <path1> <path2> ...` one-liner spanning every detected stale shell, so users can wipe leftovers from prior sessions in one shot. The reminder also now explains the typical cause (Windows file locks) so future-Gary doesn't have to re-derive it.
  - 6 unit tests pin the `worktree remove` arg parser (relative + absolute path, `--force` before/after path, non-`worktree-remove` commands, empty args). Existing 7 hygiene_reminders tests still green. Full sentinel-application lib suite: 762/762.
