## [Unreleased]

### Fixed
- **`tasks.md` shrink guard — prevent auto-block wipe from small sessions** (2026-05-16). The `task_persist` hook used to unconditionally overwrite `<repo>/tasks.md`'s `<!-- SENTINEL:TASKS:START -->` marker block with whatever the current session's `TaskList` rendered. When a brand-new session opened with only a handful of fresh tasks, the persist hook replaced an 80–100-task accumulated block with the new 3–5-task render — the user's visible task history vanished from the auto-block on the first `TaskCreate` of the new session. Root cause: `write_project_tasks_md` had no notion of "this would shrink the visible content"; it deduplicated only on byte-equality of the merged output. Fix: `write_project_tasks_md` now counts task entries in the existing block (incomplete `### [_] N. …` headers + completed `- [x] **N. …` lines) and refuses to overwrite when the new render has fewer than `SHRINK_GUARD_RATIO` (= 0.5) of the existing block's count AND the existing block has at least `SHRINK_GUARD_MIN_EXISTING` (= 10) tasks. Below the 10-task floor, small projects shrink freely (no false alarms on first-few-tasks sessions). The shrink-refuse path returns `Ok(false)` (write skipped, not a hard error) and emits a single `tracing::warn` naming both counts plus the override env var. `SENTINEL_FORCE_TASKS_MD_WRITE=1` bypasses the guard for the bulk-cleanup / project-pivot cases where shrinking is intentional. Two new helpers (`count_block_tasks`, `extract_existing_block`) and 10 new unit tests pin the behavior: counter handles open + completed + empty bodies, marker extraction returns inner body or `None`, guard refuses 80→5 wipe, guard allows 75% routine churn, guard stays off for 8-task baselines, force env overrides, initial write (no prior block) always succeeds, growth always writes through. Env-touching tests serialize on a module-local `Mutex` to avoid races in Rust's parallel test runner. Sentinel-application: 998 tests passing (988 prior + 10 new). Clippy --all-targets clean on the new code.

### Added
- **`sentinel-legatus` (C/3) — `PermissionDenied` + `Stop` hooks wired** (2026-05-17). Final commit in the series — closes the operator → consul → legatus → Claude Code → escalation → operator loop with real signals. Two existing sentinel-application hooks now fire escalations to the daemon-hosted legatus (commit B) via a tiny new `legatus_client` module. **PermissionDenied hook** (`hooks/permission_denied.rs`): on every auto-mode tool denial, fires `EscalationKind::Blocked { reason: PermissionDenied { tool } }` — operator gets a chat ping naming the denied tool. **Stop hook** (`hooks/execution_log.rs`): when a Claude Code run exits with at least one `[RUN]`/`[STEP]`/`[PHASE]` marker in the new transcript lines, fires `EscalationKind::Completed { summary: <last marker line truncated to 140 chars> }`. New `legatus_client` module reads `~/.claude/sentinel/daemon-token` (port + bearer token), spawns an OS thread that POSTs the escalation as JSON to `http://127.0.0.1:<port>/legatus/escalate` with a 3-second timeout — fire-and-forget so the hook returns immediately and never blocks Claude Code. When the daemon isn't running, the token file is absent and the thread logs at debug and exits. Workspace deps: `reqwest` gained the `blocking` feature (the legatus client is sync because hooks run synchronously and most invocations don't have a tokio runtime). sentinel-application depends on sentinel-legatus for the `EscalationKind` + `BlockReason` types. Tests: 2 new legatus_client tests (absent token returns None, fire-and-forget doesn't panic when daemon is absent); existing permission_denied + execution_log tests still pass (the fire-and-forget is invisible to sync test harnesses).

  **End-to-end runtime path now demonstrable**:
  ```
  Claude Code denies a tool
    → sentinel PermissionDenied hook fires
      → POST /legatus/escalate {"kind":"blocked", "reason":{"kind":"permission_denied","tool":"Bash"}}
        → daemon's hosted legatus task receives via channel
          → signs + sends SessionBlocked envelope over WS to consulate
            → consul-app drain task receives + EscalationEngine renders
              → SurfaceRoutingNotifier picks operator's last-seen surface
                → Telegram/Discord/Slack/WhatsApp ping arrives
  ```

- **A3 Phase 2 — `dry_run_then_commit` hook in sentinel-application** (2026-05-17). Second implementation slice of A3 (`docs/a3-dry-run-then-commit.md` §4). New `crates/sentinel-application/src/hooks/dry_run_then_commit.rs` (~460 lines) wires the Phase 1 types into a working `PreToolUse` hook. Signature: `process(input, fs, classifier, auditor) -> HookOutput` where `auditor: &dyn AuditorPort` and `classifier: &dyn ReversibilityClassifierPort` are injected. Decision pipeline: (1) Reversibility check — if class < `Irreversible`, return `allow()` (Trivially + ReversibleWithEffort are out of A3's scope; `tool_usage_gate` handles RWE). (2) Approval-marker short-circuit — if `(session_id, action_hash)` is already approved this session, return `allow()` without re-auditing. The `action_hash` is `SHA-256(tool_name || canonical(tool_input))` truncated to 16 hex chars; the new `canonicalize_json` helper recursively sorts object keys so equivalent inputs produce stable hashes regardless of caller-side serialization order. (3) Dry-run construction — extracts `_intent` / `_reasoning` / `_expected_effect` prose fields from `tool_input`; if any are blank, denies with a clear message naming the missing field (incomplete dry-runs are not worth auditor cycles). (4) Auditor call via the injected port. (5) Decision tree: `Pass` + `confidence >= 0.85` + not Catastrophic → record marker + allow; `Pass` below threshold OR class is Catastrophic → deny with "HUMAN REVIEW REQUIRED" message that names the weakest axis and the auditor's reasoning, telling the operator to invoke `hygiene_override` after reviewing; `Block { reason }` → deny with the auditor's reason inline; `AuditorError` → deny per class policy (Irreversible: "Retry when the auditor returns"; Catastrophic: "escalate to human via hygiene_override only if truly time-critical"). All deny outputs carry the `[Sentinel-Authority]` prefix via `HookOutput::deny` per the existing sentinel hook contract. Public helpers: `action_hash_for(tool, tool_input)`, `has_dry_run_approval(fs, session_id, action_hash)`, `mark_dry_run_approved(fs, session_id, action_hash)` — same shape as `tool_usage_gate`'s marker helpers, persistable across calls within a session via the temp-dir marker file pattern. 15 unit tests pin behavior: Trivially/RWE actions bypass audit silently; missing `_intent` field denies with field name; missing `session_id` denies; Irreversible + Pass + high confidence allows AND records the marker; second call with the same action short-circuits before auditor is consulted even with an erroring auditor (marker semantics verified); Pass with confidence below 0.85 triggers HUMAN REVIEW REQUIRED; Catastrophic + Pass + high confidence STILL triggers HUMAN REVIEW REQUIRED (Catastrophic always samples regardless of decision); Block decision propagates the auditor's reason; Irreversible AuditorError denies with retry message; Catastrophic AuditorError denies with explicit escalate-to-human message; `action_hash_for` is stable across JSON key reorder, varies by tool name, varies by tool input; missing tool name allows. Tests use `StaticReversibilityClassifier` (A6 Phase 3a) + an inline `StaticAuditor` mock returning canned verdicts/errors. Added `hex` as a sentinel-application dependency (was workspace-pinned but not opted-into); `sha2` was already a dep for proof-chain hashing. Module registered in `crates/sentinel-application/src/hooks/mod.rs` alphabetically between `doppler_auth0_gate` and `error_reporter`. - **A3 Phase 1 — Dry-run-then-commit domain types + `AuditorPort` trait in sentinel-domain** (2026-05-17). First implementation slice of A3 (`docs/a3-dry-run-then-commit.md`), the largest single risk-reduction primitive in the AI-factory brief and the architectural answer to the artifacts-vs-latents gap. Pure-domain value objects and a port trait — no logic yet; subsequent phases add the hook, in-memory + real adapters, and the production wiring. New `crates/sentinel-domain/src/dry_run.rs` (490 lines) defines: (a) `DryRunRequest` — the artifact the acting agent emits before any irreversible commit, with structural fields (`session_id`, `tool_name`, `tool_input`, `reversibility_class`, `constructed_at`) plus three prose fields (`agent_intent`, `agent_reasoning`, `agent_expected_effect`); builder-style `with_intent` / `with_reasoning` / `with_expected_effect` setters; `is_complete()` helper that returns `true` iff all three prose fields are non-blank (the hook short-circuits to block before paying for an LLM call when incomplete). Send + Sync per existing port convention. Module registered alphabetically in `lib.rs` between `disagreement` and `reversibility`; re-exports `AuditorAxes`, `AuditorDecision`, `AuditorError`, `AuditorVerdict`, `DryRunRequest` alongside the existing commonly-used types. - **A6 Phase 3b — `LayeredReversibilityClassifier` TOML adapter in sentinel-infrastructure** (2026-05-16). Production adapter implementing the four-layer evaluation scheme from `docs/a6-reversibility-graded-tripwires.md` §3. New `crates/sentinel-infrastructure/src/reversibility.rs` module. Layers: (1) built-in tool defaults hardcoded in `builtin_class()` (`Read`/`Glob`/`Grep`/`TaskList`/`WebFetch`/`WebSearch` → `TriviallyReversible`; `Edit`/`Write`/`TaskCreate`/`TaskUpdate` → `ReversibleWithEffort`; `Bash` delegates to L3; `mcp__*` delegates to L2; unknown → `Irreversible`); (2) per-MCP-tool defaults from `[mcp.<server>] <tool> = "Class"` TOML; (3) per-input Bash regex patterns from `[[bash.pattern]] match = "<re>" class = "..."` TOML array-of-tables — first-match wins, order preserved; (4) operator overrides from `[overrides] "<tool_name>" = "Class"` TOML — highest priority, short-circuits all earlier layers. Loader (`from_str`, `load_from_paths`) merges a sentinel-shipped defaults TOML with an optional operator-managed overrides TOML: MCP entries union (overrides win on conflict), Bash patterns concatenate with defaults FIRST so operators can add catch-all rules without losing shipped catastrophic-pattern coverage, `[overrides]` table merges. Regex patterns pre-compiled at construction; the hot path (a `classify` call) is table lookups + regex matches. Invalid regex surfaces at LOAD time, not classify time. Conservative defaults preserved at every layer: unknown MCP server/tool → `Irreversible`; Bash with no pattern match → `ReversibleWithEffort`; malformed tool name → `Irreversible`.
